

---

# 一、实验目标

验证下面这个命题：

> 对于 baseline 生成的多个 SQL 候选，使用**无训练的选择策略**，能否选出比“默认第一条输出”更好的 SQL。

---

# 二、核心思想

不训练新模型，只做这三件事：

1. **同一道题生成多个 candidate SQL**
2. **对每个 candidate 提取可计算的质量信号**
3. **用规则或同一个 7B 做“候选选择”**
4. 对比：

   * baseline 默认输出
   * rerank 后输出

---

# 三、实验设置

## 数据集选择

不要一上来跑全 dev。

建议先做一个小规模集合：

### 方案A：100 题小样本

从 Spider dev 里抽：

* 50 道 baseline 原本做对的
* 50 道 baseline 原本做错的

这样你能同时看：

* rerank 能不能救错题
* rerank 会不会伤对题

### 方案B：只看 100 道错题

更聚焦“救错题”，但看不到副作用。

我更推荐 **方案A**。

---

# 四、候选 SQL 怎么来

## 每题生成 K 个候选

建议：

* `K = 5`
* 温度适当调高一点，比如 0.6~0.8
* 或 nucleus sampling

目标是让模型输出**有差异的多个候选**，而不是 5 个几乎一样的 SQL。

---

## 你要保存什么

每道题存：

* `question`
* `db_id`
* `gold_sql`
* `candidate_sqls = [sql1, sql2, ..., sql5]`

如果能一起保存：

* 候选是否执行成功
* 执行结果行数
* execution accuracy
  会更方便后续分析。

---

# 五、先做最简单的三种“无训练选择器”

你不要一开始搞复杂。
先设计 3 个 selector，逐级增强。

---

## Selector 1：Execution-first

规则：

1. 优先保留**能执行**的 SQL
2. 如果多个都能执行，优先选择：

   * 非空结果
   * 结构更完整者（可先简单用长度或 clause 数量近似）

### 目的

验证“execution + 非空”这种最弱信号有没有用。

### 预期

我猜帮助有限，但它是 baseline。

---

## Selector 2：Schema-validity + Execution

给每个候选打简单分：

### 分数构成示例

* +3：能执行
* +2：结果非空
* +1：所有表列都在 schema 中
* +1：没有明显重复/畸形 SQL
* -1：明显缺失 SELECT/FROM/ORDER/GROUP 等关键结构
* -1：引用未知列
* -1：语法可执行但结构可疑（例如 group/order 不匹配，可以先做很弱的检测）

最后选分最高的。

### 目的

验证：**纯规则质量信号**能不能比 baseline 更好。

---

## Selector 3：同模型判别（不训练）

还是用你当前 7B，但不让它生成新 SQL，只让它**在候选里选最优**。

### 输入形式

给模型：

* question
* schema
* 候选 SQL 1~5

让它回答：

> 哪一个最有可能正确回答问题？只输出编号。

### 很重要

不要让它解释太多，避免输出发散。
直接要求：

* 输出 `1/2/3/4/5`
* 或输出 JSON：`{"best": 3}`

### 目的

验证：**同一个模型是否具备“判断优于生成”的能力**

这一步很关键。

---

# 六、推荐的具体实验对比组

你至少做下面 4 组：

## Baseline

* 取第 1 个生成结果
* 或你当前系统默认输出

## Group A

* 从 5 个候选中，选“第一个能执行且非空的”

## Group B

* 用规则打分选最优

## Group C

* 用同模型判别选最优

---

# 七、评测指标

最核心的是：

## 1. Execution Accuracy

对比各组最终选出的 SQL 的 EX 准确率。

---

## 2. Oracle Upper Bound

这非常重要。

对每道题 5 个候选，如果**其中任意一个**是对的，就记为 oracle success。

也就是算：

> **如果我有完美 reranker，这 5 个候选最多能达到多少分？**

这个指标决定了这条路线是否值得继续。

---

## 3. Candidate Diversity

统计：

* 每题 5 个候选里平均有多少个不同 SQL
* 有多少题 5 个候选几乎完全一样

如果几乎都一样，那 rerank 本身就没空间。

---

## 4. Rescue Rate

在 baseline 错题里：

> rerank 选中了正确 SQL 的比例

---

## 5. Regression Rate

在 baseline 对题里：

> rerank 把本来对的选坏了多少

---

# 八、最关键的一张表

最后你应该得到类似这样一张表：

| 方法                | EX Acc | Rescue Rate | Regression Rate |
| ----------------- | -----: | ----------: | --------------: |
| baseline          |      x |           - |               - |
| execution-first   |      x |           x |               x |
| rule-based        |      x |           x |               x |
| same-model rerank |      x |           x |               x |
| oracle@5          |      x |           - |               - |

这张表会直接告诉你：

* rerank 有没有潜力
* 潜力来自哪里
* 值不值得继续做训练版判别器

---

# 九、如何解释实验结果

## 情况1：oracle@5 明显高，但无训练 rerank 提升很小

这说明：

> 候选里其实有正确答案，但你不会挑。

这时就值得继续做判别器/训练 reranker。

---

## 情况2：oracle@5 本身就几乎不高

这说明：

> 生成阶段根本没采到更好的候选。

这时 rerank 路线不值得继续，应该回去增强 generation。

---

## 情况3：same-model rerank 比规则强很多

这说明：

> 模型“判断”能力强于“第一次生成”能力

这时很值得继续深入。

---

## 情况4：same-model rerank 也没用

这说明：

> 候选差异不可分，或者模型没有稳定判别能力

那这条路大概率收益有限。

---

# 十、如何控制成本

你担心单人项目成本，这个实验可以压得很低：

## 最小版配置

* 100 道题
* 每题 5 个候选
* 共 500 条 SQL 候选
* 只做 3 个 selector
* 不训练任何模型

这完全是单人可承受的。

---

# 十一、我建议的实施顺序

## 第一步

先只做：

* baseline
* oracle@5

因为这一步先回答最关键的问题：

> **候选里到底有没有值得挑的东西？**

如果 oracle@5 比 baseline 只高 1 点以内，那后面都不用做了。

---

## 第二步

如果 oracle@5 有明显空间，再做：

* execution-first
* rule-based

---

## 第三步

如果规则版仍然没吃到 oracle 空间，再做：

* same-model rerank

---

# 十二、same-model rerank 的 prompt 建议

你可以用这种最简 prompt：

**System**

```text
You are evaluating SQL candidates for a Text-to-SQL task.
Choose the single SQL that is most likely to correctly answer the question under the given schema.
Output only the candidate number.
```

**User**

```text
Question:
...

Schema:
...

Candidate 1:
...

Candidate 2:
...

Candidate 3:
...

Candidate 4:
...

Candidate 5:
...
```

要求只输出：

```text
3
```

不要让它长篇解释。

---

# 十三、如果你想再稳一点

你还可以加一个 very cheap 的 trick：

先过滤掉：

* 不可执行
* 空结果
* 明显引用不存在列的

然后再让 same-model 在剩下候选中选。

这样判别任务更简单。

---

# 十四、这项实验的价值

这个最小实验的意义不是“马上涨分”，而是帮你做路线决策：

* **oracle@5 高** → rerank 值得继续
* **oracle@5 低** → rerank 不值得继续
* **same-model rerank 有效** → 可考虑更深入
* **same-model rerank 无效** → 不要再投入

也就是说，这个实验是一个**方向验证器**。

---

# 十五、我建议你现在就做的最终版本

最小可行版本：

1. 从 dev 抽 100 题
2. 每题采样 5 个 SQL
3. 计算 baseline 与 oracle@5
4. 如果 oracle@5 有空间，再做：

   * execution-first
   * rule-based
   * same-model rerank
