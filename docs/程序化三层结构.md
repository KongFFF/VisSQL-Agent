# 一、程序化中间层是什么

## 1. 本质定义

程序化中间层，就是：

> **不让模型直接输出最终 SQL，而是先输出一个“程序可以理解、检查、操作”的结构化表示。**

这个表示可以是：

* JSON
* dict
* dataclass
* AST风格对象

但重点不是格式，而是：

> 它必须是**稳定的、字段明确的、能被代码读取的**

---

## 2. 为什么需要它

因为直接生成 SQL 有几个问题：

* 小模型容易漏条件
* alias 容易乱
* group by / order by / having 容易不一致
* 你很难知道它究竟“想错在哪一步”

而如果先输出中间层，你就能把复杂问题拆成一组更小的决策：

* 选哪些列
* 主表是谁
* 有没有 join
* join key 是什么
* 过滤条件是什么
* 排序指标是什么
* limit 是多少

这时候模型不再“写程序”，而是在“填表单”。

---

## 3. 你这个项目里的中间层长什么样

比如 top-k 题：

> What is the airport code of the airport with the most flights?

不要直接让模型写：

```sql
SELECT T1.AirportCode ...
```

而是让它先输出：

```json
{
  "pattern": "topk",
  "subtype": "group_topk",
  "select_columns": ["AIRPORTS.AirportCode"],
  "base_table": "AIRPORTS",
  "joins": [
    {
      "table": "FLIGHTS",
      "on": [
        "AIRPORTS.AirportCode = FLIGHTS.DestAirport",
        "AIRPORTS.AirportCode = FLIGHTS.SourceAirport"
      ],
      "join_logic": "or"
    }
  ],
  "where_conditions": [],
  "group_by": ["AIRPORTS.AirportCode"],
  "order_metric": {
    "agg": "count",
    "column": "*"
  },
  "order_direction": "DESC",
  "limit": 1
}
```

这就是程序化中间层。

---

## 4. 程序化中间层的核心价值

### 第一，降低任务难度

模型不用一次性生成完整 SQL，只要把槽位填对。

### 第二，方便错误定位

如果 SQL 错了，你能知道是：

* select 错了
* group_by 错了
* order_metric 错了

而不是只能看到“整条 SQL 不对”。

### 第三，方便程序接手

后面的 validator 和 builder 都必须基于这个中间层工作。

---

## 5. 你该如何设计中间层

对每种 pattern 单独定义 schema。

比如：

---

### Top-K 的中间层

```json
{
  "pattern": "topk",
  "subtype": "simple_topk | group_topk",
  "select_columns": [],
  "base_table": "",
  "joins": [],
  "where_conditions": [],
  "group_by": [],
  "order_metric": {
    "agg": "",
    "column": ""
  },
  "order_direction": "ASC | DESC",
  "limit": 1
}
```

---

### Nested 的中间层

```json
{
  "pattern": "nested",
  "subtype": "comparison_nested",
  "outer": {
    "select_columns": [],
    "base_table": "",
    "joins": [],
    "where_prefix_conditions": [],
    "compare_column": "",
    "compare_op": ">"
  },
  "inner": {
    "agg": "avg",
    "column": "",
    "base_table": "",
    "joins": [],
    "where_conditions": []
  }
}
```

---

### Set 的中间层

```json
{
  "pattern": "set",
  "subtype": "difference_set | intersection_set",
  "left": {
    "select_columns": [],
    "base_table": "",
    "joins": [],
    "where_conditions": [],
    "key_column": ""
  },
  "right": {
    "select_columns": [],
    "base_table": "",
    "joins": [],
    "where_conditions": [],
    "key_column": ""
  },
  "set_operator": "NOT IN | INTERSECT | EXCEPT"
}
```

# 二、程序化约束是什么

## 1. 本质定义

程序化约束，就是：

> **模型给出中间层后，由代码判断它是否满足 SQL 结构规则、schema 规则和 pattern 规则。**

也就是：

> 模型负责提议
> 程序负责审核

---

## 2. 为什么必须有程序化约束

如果你没有这层，那么“中间层”只是换了个皮的 prompt 输出。

例如模型输出：

```json
{
  "pattern": "topk",
  "subtype": "group_topk",
  "select_columns": ["AIRLINES.Abbreviation", "AIRLINES.Country"],
  "base_table": "AIRLINES",
  "group_by": [],
  "order_metric": {
    "agg": "count",
    "column": "*"
  },
  "limit": 1
}
```

这显然不对，因为：

* 既然是 group_topk，就应该有 group_by
* 只有 AIRLINES 主表，但 count 的对象其实来自 flights
* 可能还缺 join

如果没有程序化约束，这种输出你也只能硬着头皮拼 SQL，然后得到垃圾结果。

---

## 3. 程序化约束分哪几类

我建议你至少做 3 类约束。

---

## 第一类：字段完整性约束

检查该有的字段是否存在。

例如：

* topk 必须有 `order_direction`
* group_topk 必须有 `group_by`
* nested 必须有 `outer.compare_column`
* set 必须有 `left.key_column` 和 `right.key_column`

这是最基础的。

---

## 第二类：schema 合法性约束

检查字段值是否真的在数据库 schema 里存在。

例如：

* `AIRLINES.Abbreviation` 这个列是否存在
* `FLIGHTS.Airline` 是否存在
* `concert.year` 是否存在

再进一步：

* join 的两边列是不是可连接
* key 类型是否大致匹配

---

## 第三类：结构一致性约束

这是最重要的，也是最像“方法”的一层。

例如：

### 对 Top-K：

* simple_topk 不应该有 group_by
* group_topk 必须有 group_by
* group_topk 的 `order_metric.agg` 不能为空
* 如果有 group_by，select 里的非聚合列必须可由 group_by 支撑

### 对 Nested：

* inner 必须返回单值
* inner 最好是聚合表达式
* outer.compare_column 与 inner 返回值类型要兼容

### 对 Set：

* left/right 的 key_column 必须可对齐
* 返回列应该来自正确一侧
* 左右集合的逻辑不能反

---

## 4. 程序化约束具体长什么样

例如 Top-K 的 Python 伪代码：

```python
def validate_topk_slots(slots, schema):
    errors = []

    if slots["subtype"] == "group_topk":
        if not slots.get("group_by"):
            errors.append("group_topk_missing_group_by")

        metric = slots.get("order_metric", {})
        if not metric.get("agg"):
            errors.append("group_topk_missing_agg")

    if slots["subtype"] == "simple_topk":
        if slots.get("group_by"):
            errors.append("simple_topk_should_not_have_group_by")

    for col in slots.get("select_columns", []):
        if not schema.has_column(col):
            errors.append(f"unknown_select_column:{col}")

    return errors
```

这就是程序化约束。

---

## 5. 程序化约束的作用

### 第一，阻止明显错误进入 SQL builder

避免“错上加错”。

### 第二，为 repair 提供精确信号

你可以把错误信息发回模型：

* 缺少 group_by
* 列不存在
* subtype 与结构不一致

这样 repair 比“请重写 SQL”稳定得多。

### 第三，让系统不完全依赖 prompt

这就是你摆脱“纯 prompt 工程”的关键。

# 三、程序化合成是什么

## 1. 本质定义

程序化合成，就是：

> **模型不负责写最终 SQL，由代码根据合法的中间层模板化生成 SQL。**

也就是：

* 模型决定“内容”
* 代码决定“形式”

---

## 2. 为什么需要它

因为 SQL 的最终表面形式很脆弱：

* 括号
* 别名
* join 顺序
* where/group/order 的拼接顺序
* limit 放哪
* 聚合函数怎么写

这些地方很机械，但很容易错。

所以最好的策略不是让模型“又思考又写语法”，而是：

> 让模型只思考
> 让程序负责语法

---

## 3. 程序化合成如何工作

以 Top-K 为例。

如果中间层是：

```json
{
  "pattern": "topk",
  "subtype": "group_topk",
  "select_columns": ["AIRLINES.Abbreviation", "AIRLINES.Country"],
  "base_table": "AIRLINES",
  "joins": [
    {
      "table": "FLIGHTS",
      "on": ["AIRLINES.uid = FLIGHTS.Airline"]
    }
  ],
  "where_conditions": [],
  "group_by": ["FLIGHTS.Airline"],
  "order_metric": {
    "agg": "count",
    "column": "*"
  },
  "order_direction": "ASC",
  "limit": 1
}
```

程序就能拼出：

```sql
SELECT AIRLINES.Abbreviation, AIRLINES.Country
FROM AIRLINES
JOIN FLIGHTS ON AIRLINES.uid = FLIGHTS.Airline
GROUP BY FLIGHTS.Airline
ORDER BY count(*) ASC
LIMIT 1
```

---

## 4. 你应该怎么写 builder

建议每个 pattern 一个 builder。

例如：

* `build_topk_sql(slots)`
* `build_nested_sql(slots)`
* `build_set_sql(slots)`

---

### Top-K builder 的逻辑

#### simple_topk

按顺序拼：

1. SELECT
2. FROM
3. JOIN
4. WHERE
5. ORDER BY raw column
6. LIMIT

#### group_topk

按顺序拼：

1. SELECT
2. FROM
3. JOIN
4. WHERE
5. GROUP BY
6. ORDER BY aggregation
7. LIMIT

---

### Nested builder 的逻辑

1. 先拼 inner query
2. 再把 inner query 填到 outer where compare 中

例如：

```sql
WHERE age > (SELECT avg(age) FROM singer)
```

这里 inner query 完全可以先单独函数生成。

---

### Set builder 的逻辑

1. 先拼 left
2. 再拼 right
3. 根据 subtype 选择：

* `NOT IN`
* `INTERSECT`
* `EXCEPT`

或者统一变成 `NOT EXISTS`

---

## 5. 程序化合成的最大价值

### 第一，输出稳定

同样的中间层总是生成同样的 SQL 结构。

### 第二，便于调试

你一看中间层就知道 builder 应该如何表现。

### 第三，便于做 ablation

你可以单独比较：

* 直接生成 SQL
* 结构化中间层 + builder

这就是实验点。

---

# 四、三者之间的关系

你可以把它们理解成工厂流水线：

---

## 第一步：程序化中间层

把自然语言问题变成结构化计划。

例如：

> 这是一个 group_topk
> 返回 abbreviation, country
> 按 count(*) 升序
> limit 1

---

## 第二步：程序化约束

检查这个计划合不合理。

例如：

* 有没有 group_by
* 列是否存在
* join 是否合法

---

## 第三步：程序化合成

把合理计划翻译成 SQL。

例如：

```sql
SELECT ...
FROM ...
GROUP BY ...
ORDER BY ...
LIMIT 1
```

---

# 五、如果缺了其中一个，会怎样

## 只做中间层，不做约束

那中间层只是“看起来更整齐”，但不可靠。

## 只做约束，不做中间层

那你没东西可检查，还是直接靠模型吐 SQL。

## 只做中间层和约束，不做合成

那最终 SQL 还是模型写，稳定性提升有限。

---

# 六、放到你项目里，最实际的落地方式

你现在最适合这样做：

---

## Top-K

先做完整三件套：

### 程序化中间层

定义 `simple_topk / group_topk` 的 JSON schema

### 程序化约束

检查：

* subtype 合法
* group_by 合法
* order_metric 合法
* select/group 一致性

### 程序化合成

写 `build_topk_sql()`

---

## Nested

先做一类最简单的：

* `comparison_nested`

同样三件套。

---

## Set

先做：

* `difference_set`

同样三件套。

---

# 七、你可以怎么向别人解释这套东西

你可以这样说：

> 我们没有让小模型直接生成 SQL，而是先生成结构化中间表示，再通过程序规则进行合法性校验，最后由模板化 SQL composer 输出最终 SQL。

这句话比“我写了几个 prompt”要强很多。

---

# 八、最简短的压缩总结

### 程序化中间层

把“写 SQL”改成“填结构化槽位”。

### 程序化约束

把“模型说了算”改成“模型提议，程序审核”。

### 程序化合成

把“模型写语法”改成“程序按模板生成 SQL”。
