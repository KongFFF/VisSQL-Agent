# VisSQL-Agent 保研面试故事线

> 这是一份**用于保研面试/简历讲述的包装版故事线**，不要求与真实开发顺序完全一致。  
> 目标不是复盘所有试错，而是讲清楚：**发现了什么问题、为什么这样设计、做了哪些关键取舍、最后系统为什么有效。**

## 一句话版本

我做的是一个面向 Spider 的 **Text-to-SQL 智能代理系统**。  
项目从一个端到端微调模型出发，逐步加入 **执行反馈、检索增强、值级 grounding、语义风险控制、保守列级检索、seed-first 子图保留、可解释 retrieval**，最终把执行准确率提升到 **79.4%**，并且在过程中系统性分析了多条失败路线，最后收敛出一条更稳定的方案。

---

## 建议采用的总叙事

建议把整个项目讲成 **五个阶段**，不要按原始版本号逐个念。

### 阶段 1：建立一个可靠的端到端基线

**核心问题**

最开始我先验证一个最直接的思路：  
把 Spider 数据集整理成模型更容易理解的 schema + question -> SQL 训练格式，在 Qwen2.5-Coder-7B-Instruct 上做 LoRA 微调，先拿到一个可以稳定输出 SQL 的基线。

**关键工作**

- 重构 Spider 的 schema 表达方式
- 统一 system prompt 和训练格式
- 用 LoRA 做端到端 Text-to-SQL 微调

**阶段性结论**

这一阶段让我拿到了一个大约 **71%** 的基线。  
它说明模型已经具备了基本的 SQL 生成能力，但也暴露出一个很明显的问题：

- 模型会写 SQL
- 但对复杂 schema、跨表关系和值条件的处理并不稳定

**这一阶段怎么讲**

不要讲成“我调了很多微调参数”。  
要讲成：

> 我先搭建了一个可复现的 direct-SQL baseline，确认模型本身具备基础语义到 SQL 的映射能力，然后把后续工作聚焦在系统层补强，而不是盲目继续堆训练。

---

### 阶段 2：从单轮生成转向 Agent 化执行闭环

**核心问题**

直接生成 SQL 的模型，常见错误有两类：

- 语法对不上数据库真实 schema
- SQL 能生成，但执行失败或结果为空

所以我把系统从“单轮生成器”升级成“可执行代理”：

- 先生成 SQL
- 再执行 SQL
- 利用执行反馈做自修复

**关键工作**

- 搭建 ReAct/Reflexion 风格的主链路
- 把 SQL 执行器接入 agent
- 加入错误反馈和重试机制
- 补上基础的 schema-aware retrieval

**阶段性结论**

这一阶段把系统从“会写 SQL”推进成了“会基于数据库反馈修正 SQL”，整体准确率提升到 **约 75%**。

但我也发现：

- 仅靠执行报错做修复，能解决“语法错”
- 对“能执行但语义错”的题帮助有限

这为后面的误差分析指明了方向。

**这一阶段怎么讲**

重点突出“系统形态升级”：

> 我没有把问题停留在 prompt 或微调上，而是把 Text-to-SQL 做成了一个可执行的 agent：模型先提出 SQL 假设，数据库执行器给出真实环境反馈，系统再做迭代修复。

---

### 阶段 3：误差分析后，发现主瓶颈不是找表，而是值级 grounding

**核心问题**

在这一阶段，我对错误样本做了专门分析，发现一个重要现象：

- 相关表大多数时候其实已经找到了
- 真正的大头错误，出现在 **值链接、实体消歧、条件落列错误**

例如：

- `USA` 该落在代码列还是名称列
- `APG` 是机场代码还是机场名称
- `IsOfficial` 真值是 `T/F` 还是 `Yes/No`

这说明系统已经不是“找不到表”，而是“看到了表，但没把自然语言里的值正确落到数据库字段上”。

**关键工作**

- 在 retrieval 后增加 value hints
- 为候选列补充 sample values
- 显式给出 question entity matches
- 强化 code/name、枚举值、布尔值等值级线索

**阶段性结论**

这是项目里第一次比较大的跃升，执行准确率提升到 **78.5%**。

这一步是整个项目的重要分水岭，因为它证明了：

> 在 Text-to-SQL 后半程，真正关键的不是再多找几张表，而是做 schema-aware 之外的 **value-aware grounding**。

**这一阶段怎么讲**

这是你最值得讲的“洞察”之一。  
建议直接说：

> 我通过错题分析发现，系统的主瓶颈不是表级召回，而是值级 grounding。于是我把 schema prompt 从“只给结构”升级成“结构 + 值线索”，这一步带来了最显著的单阶段提升。

---

### 阶段 4：从“执行成功”升级到“语义风险可控”

**核心问题**

值级 grounding 做完以后，系统还有一个残留问题：

- 有些 SQL 能执行
- 但条件挂错列、group 粒度错、集合语义错
- 也就是说，“执行成功”不等于“语义正确”

所以我尝试加入 semantic verifier。

**这里一定要保留一个负结果**

我做的第一版 verifier 并没有直接提升，反而一度掉分。  
原因是：

- 规则太激进
- 会把一些本来正确的 SQL 误判为高风险
- 然后强行重写，导致把对答案改坏

这个负结果非常值得讲，因为它体现了你不是“盲目堆模块”，而是真的做了归因分析。

**我后来的修正**

- 去掉误报很高的 `missing_group_by`
- 只保留更可靠的风险检查
- 加入 fallback 机制：始终保留第一条执行成功的 SQL
- 避免 verifier 把已经正确的答案改坏

**阶段性结论**

修正后系统稳定回升到 **78.9%**。

**这一阶段怎么讲**

建议这样讲：

> 我发现“执行成功”不是充分条件，于是加入了语义级 verifier。但第一版 verifier 因为过度干预反而伤分。后面我保留高价值风险信号，并增加成功候选回退机制，把它从负资产修成了正资产。这体现了我对 agent 系统稳定性的关注，而不是单纯追求更复杂的模块。

---

### 阶段 5：从激进列级检索收敛到保守列级检索 + seed-first 子图保留

**核心问题**

到了这一阶段，系统剩下的很多错题都和“字段级语义定位”和“检索子图截断”有关。

我一开始尝试了激进的列级检索：

- 给每题大规模加入“优先关注字段”
- 让模型更多参考列级命中

结果反而掉分。  
原因是 direct-SQL 微调模型会被这些强提示带偏，出现：

- 多余 join
- 同名列误导
- 在同一批表里选错列

**这里第二个负结果也很值得保留**

不是“列级检索没用”，而是：

> 激进列提示会把一个已经会直接写 SQL 的模型，带入新的 prompt 分布，反而伤害稳定性。

**我后来的修正**

- 对列命中做歧义抑制
- 对泛 `id/code/fk` 列降权
- 只在高置信时展示极少量列提示
- 把列信号从“强 steering”改成“保守辅助”

这一步把系统提升到 **79.1%**。

然后我进一步分析发现：

- 真正决定后续生成质量的，不只是“有没有找到相关表”
- 还有“相关 seed 表在 top-k schema 子图里会不会被截断掉”

于是我加入 **seed-first retention**：

- 优先保住高价值 seed tables
- 再补充其余支持表
- 避免关键表被 top-k 截断

这一步把系统进一步提升到 **79.4%**。

最后我又补了一层 **retrieval explanation**，让系统不仅输出结果，还能解释：

- 为什么选这几张表
- 哪些列和值参与了检索
- 哪些候选因为 top-k 被丢掉
- 当前 retrieval 有哪些歧义

**最终结论**

项目最终收敛成一个：

- direct-SQL 微调为基础
- agent 执行为骨架
- value grounding 为关键增益点
- verifier 为稳定性控制
- 保守列级检索与 seed-first 为检索增强
- explanation 为可解释层

的数据库智能代理系统，最终执行准确率达到 **79.4%**。

## 推荐你在面试里强调的三个“研究味”结论

### 1. 主瓶颈迁移：从 schema-level 到 value-level

这是项目里最强的洞察之一。  
你不是一直在调 retrieval，而是通过错题分析发现：

- 前期瓶颈是表和关系
- 后期瓶颈转移到了值级 grounding

这个非常像科研里的“误差主导项转移”。

### 2. 不是所有结构增强都会涨分，分布匹配很重要

这个结论来自两个负结果：

- 激进 verifier 会伤分
- 激进列级提示会伤分

你可以把它上升成一句很好的方法论：

> 对已经做过 direct-SQL 微调的模型，系统增强必须尊重模型原有分布，不能用过强的中间约束去破坏它的稳定生成习惯。

### 3. 子图保留策略和检索解释同样重要

最后这一阶段的亮点不只是提分，而是你发现：

- 有时问题不是没找到表
- 而是找到了关键表，却在 top-k 截断时丢掉了

这说明你关注的不是“是否命中”，而是“命中后能否被稳定保留并有效利用”。

这个点非常适合往“图驱动/可解释 retrieval”方向去讲。

## 不建议在面试里按原样展开的内容

下面这些内容可以保留在文档里，但不建议在面试主线里细讲：

- v1 到 v6 每个微调版本的细节参数差异
- 候选池路线的所有 selector 细节
- phase1/phase2 各个模板版本的完整枚举
- bridge v1 的实现细枝末节

这些内容的问题不是“没价值”，而是：

- 细节太散
- 容易削弱主线
- 面试官很难快速抓住你真正的贡献

建议处理方式：

- 可以提“我还探索过候选池和结构模板路线，但主线最终收敛到 value grounding + risk-controlled retrieval”
- 不要展开成流水账

## 3 分钟口述版

我这个项目是一个面向 Spider 的 Text-to-SQL 智能代理系统。最开始我先在 Qwen2.5-Coder-7B-Instruct 上做了基于 Spider 的 direct-SQL LoRA 微调，拿到了大约 71% 的基线。然后我发现只靠单轮生成不够稳定，所以把系统做成了 agent 形态：模型先生成 SQL，再由数据库执行器返回真实反馈，系统根据执行结果做修正，这一步把性能推进到了 75% 左右。

后面我对错误样本做了系统分析，发现主瓶颈不是找不到相关表，而是值级 grounding 不够好，比如代码值、枚举值、布尔值和 name/code 列经常落错。于是我在 retrieval 后加入了 value hints 和 sample values，把 schema prompt 从“只给结构”升级成“结构 + 值线索”，这一阶段提升最明显，准确率到了 78.5%。

再往后，我发现“执行成功”不等于“语义正确”，所以加了 semantic verifier。但第一版 verifier 过度干预，反而掉分。后来我保留高价值风险信号，同时加了 fallback，保证已经成功的 SQL 不会被误改坏，系统回升到 78.9%。最后我继续做了保守列级检索和 seed-first 子图保留，让检索既能利用列级语义，又不会把 direct-SQL 模型带偏，最终做到 79.4%。同时我还加了 retrieval explanation，让系统能够解释为什么选这些表和列，这样项目不只是一个 SQL 生成器，而是一个可解释的数据库智能代理。

---

## 你可以主动说出的亮点

- 我不是只做了一个 baseline，而是做了完整的误差分析和系统收敛。
- 我保留了失败实验，并用它们反过来指导设计收敛。
- 我发现了项目中“主瓶颈迁移”的过程：从 schema-level 到 value-level。
- 我最终做出的不是单纯的 prompt 工程，而是一个带执行、检索、风险控制和解释层的 agent 系统。

## 最后建议

如果你把这份故事线用于简历或面试，建议始终坚持一个原则：

> **讲“为什么这样设计”，胜过讲“我一共改了多少版本”。**

你最强的地方，不是做了很多 patch，而是你能把这些 patch 总结成一条清晰的系统演化逻辑。
